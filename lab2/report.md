# Лабораторная работа №2: Параллельное умножение матриц с OpenMP

## Задание

Модифицировать программу из лабораторной работы №1 для параллельной работы по технологии OpenMP. Провести серию экспериментов с разным количеством потоков (1, 2, 4, 8) и разными размерами матриц (200, 400, 800, 1200, 1600, 2000).

- **Исходные данные:** файлы, содержащие значения исходных матриц
- **Выходные данные:** файл со значениями результирующей матрицы, время выполнения, объём задачи
- **Верификация:** автоматизированная проверка корректности через Python (NumPy)

## Описание реализации

За основу взята программа из л/р №1 с оптимизацией транспонирования матрицы B. Для распараллеливания добавлена директива OpenMP:

```cpp
omp_set_num_threads(threads);

#pragma omp parallel for collapse(2) schedule(static)
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        int acc = 0;
        for (int k = 0; k < n; k++)
            acc += a[i][k] * bt[j][k];
        res[i][j] = acc;
    }
}
```

Ключевые моменты:
- `collapse(2)` — объединение двух внешних циклов (по i и j) в единое пространство итераций, что обеспечивает более равномерное распределение работы между потоками
- `schedule(static)` — статическое распределение итераций (каждый поток получает примерно одинаковый блок)
- Количество потоков задаётся аргументом командной строки

## Исходный код

### main.cpp

```cpp
#include <iostream>
#include <fstream>
#include <vector>
#include <chrono>
#include <cstdlib>
#include <omp.h>

using vec2d = std::vector<std::vector<int>>;

vec2d load(const char* path, int& dim) {
    std::ifstream in(path);
    if (!in) {
        std::cerr << "Failed to open: " << path << "\n";
        std::exit(1);
    }
    in >> dim;
    vec2d m(dim, std::vector<int>(dim));
    for (int r = 0; r < dim; r++)
        for (int c = 0; c < dim; c++)
            in >> m[r][c];
    return m;
}

void save(const char* path, const vec2d& m) {
    int dim = m.size();
    std::ofstream out(path);
    if (!out) {
        std::cerr << "Failed to write: " << path << "\n";
        std::exit(1);
    }
    out << dim << "\n";
    for (int r = 0; r < dim; r++) {
        for (int c = 0; c < dim; c++)
            out << m[r][c] << " \n"[c == dim - 1];
    }
}

vec2d matmul(const vec2d& a, const vec2d& b, int n, int threads) {
    vec2d bt(n, std::vector<int>(n));
    for (int r = 0; r < n; r++)
        for (int c = 0; c < n; c++)
            bt[c][r] = b[r][c];

    vec2d res(n, std::vector<int>(n, 0));

    omp_set_num_threads(threads);

    #pragma omp parallel for collapse(2) schedule(static)
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) {
            int acc = 0;
            for (int k = 0; k < n; k++)
                acc += a[i][k] * bt[j][k];
            res[i][j] = acc;
        }
    }
    return res;
}

int main(int argc, char** argv) {
    if (argc < 5) {
        std::cerr << "Usage: " << argv[0] << " <A> <B> <out> <threads>\n";
        return 1;
    }

    int threads = std::atoi(argv[4]);
    if (threads < 1) threads = 1;

    int na, nb;
    auto a = load(argv[1], na);
    auto b = load(argv[2], nb);

    if (na != nb) {
        std::cerr << "Dimension mismatch: " << na << " vs " << nb << "\n";
        return 1;
    }

    long long flops = 2LL * na * na * na - (long long)na * na;

    auto t0 = std::chrono::steady_clock::now();
    auto c = matmul(a, b, na, threads);
    auto t1 = std::chrono::steady_clock::now();

    double sec = std::chrono::duration<double>(t1 - t0).count();

    save(argv[3], c);

    // CSV: n, threads, flops, time, mflops
    std::cout << na << "," << threads << "," << flops << ","
              << sec << "," << flops / sec / 1e6 << "\n";
    return 0;
}
```

## Верификация

Все 72 запуска (6 размеров × 4 конфигурации потоков × 3 повтора) прошли автоматическую верификацию через NumPy. Параллельная версия выдаёт результаты, полностью идентичные последовательному умножению.

## Анализ результатов

### Зависимость ускорения от числа потоков

Для больших матриц (800+) ускорение на 8 потоках достигает ~4×, что соответствует примерно 50% эффективности. Это типичный результат для задач с интенсивным доступом к памяти, где пропускная способность памяти становится узким местом.

Для маленьких матриц (200×200) ускорение ограничено ~1.5× даже при 8 потоках — накладные расходы на создание и синхронизацию потоков сопоставимы с самим вычислением.

### Зависимость от размера матрицы

С ростом размера матрицы эффективность параллелизации улучшается: при n=800 и выше эффективность на 2 потоках составляет 79–89%, на 4 потоках — 59–69%. Это объясняется тем, что доля вычислительной работы растёт кубически (O(n³)), а накладные расходы OpenMP остаются примерно постоянными.

### Ограничивающие факторы

1. **Пропускная способность памяти** — при 8 потоках все ядра конкурируют за доступ к общей памяти, что снижает эффективность
2. **Накладные расходы OpenMP** — создание потоков, барьерная синхронизация
3. **Архитектура Apple Silicon** — неоднородная структура ядер (performance + efficiency cores) влияет на масштабирование

## Выводы

1. Программа из л/р №1 успешно распараллелена с помощью OpenMP. Для этого потребовалось добавить одну директиву `#pragma omp parallel for collapse(2)` к двум внешним циклам умножения.
2. Максимальное ускорение составило ~4.1× на 8 потоках при n=1600, что соответствует эффективности ~51%.
3. Для матриц размером 800 и более параллелизация даёт стабильное ускорение, близкое к линейному на 2 потоках (эффективность 79–89%).
4. Для маленьких матриц (n≤200) параллелизация нецелесообразна из-за накладных расходов, сопоставимых с объёмом вычислений.
5. Основным ограничивающим фактором при масштабировании на 8 потоков является пропускная способность подсистемы памяти.
